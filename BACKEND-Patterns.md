## Stack

Python 3.11+ · FastAPI · Pydantic v2 · SQLAlchemy (simple single-table CRUD) + raw SQL (JOINs / aggregations / window functions) · Alembic (all migrations) · httpx (external calls, gateways only). SQLite in dev → PostgreSQL in prod. Single full-stack repo: `/backend` (FastAPI) + `/frontend` (React/Vite); FastAPI serves the built SPA + the JSON API.

---

## The ten principles

1. No business logic in routes — routes wire HTTP to queries/engine and return.
2. Schemas (Pydantic API contracts) are not models (SQLAlchemy DB structure) — never return a model from a route.
3. SQL for complex reads (JOINs, aggregations, windows); ORM for simple CRUD — never mix the two in one function.
4. Engines own domain logic; queries own data access; gateways own external calls.
5. Filter, sort, and paginate in SQL, not in Python.
6. Dependency injection via `Depends` (DB session, config; current user later).
7. Alembic owns every schema change — never `create_all()` in prod.
8. Review autogenerated migrations — they miss raw-SQL tables and data migrations.
9. Never edit an applied migration — write a new one.
10. Config via `.env`; secrets never in code.

## Service shape (per domain)

```
backend/
  app/
    main.py          ← FastAPI app, router registration, static-serving, startup
    core/            ← config (.env), db session, shared deps
    <domain>/
      routes.py      ← HTTP only; no business logic
      queries.py     ← data access (SQL + simple ORM); takes a session param
      engine/        ← domain computation, when needed
      schemas.py     ← Pydantic request/response contracts
  alembic/           ← migrations (baseline empty at 001)
```

A service may import another service's `queries` (read-only), never its `routes` or `engine`; shared logic → `core/`.

## API design

- Routes under `/api/{domain}/{resource}`.
- Envelope: `{ "data": ... }` on success (lists add `"meta": { total, page, per_page }`); `{ "error": { "code", "message" } }` on failure.
- List endpoints accept `page`, `per_page`, filter params, `sort_by`, `sort_dir` — applied in SQL.

---

## Database & migrations

- SQLAlchemy models per domain; Alembic for all schema changes.
- Baseline (empty) migration at 001; `alembic upgrade head` runs before app start (entrypoint).
- SQLite in dev, Postgres in prod — keep schema portable (no engine-specific types without need).
- No business tables until a piece declares them: 001 has none; the first business table arrives at 008 (the contact/quote store-and-forward submissions table).

---

## Serving the frontend

- FastAPI serves the Vite build: `StaticFiles` for assets + a catch-all route returning the built `index.html`, so client-side React Router resolves `/en` `/ru` paths.
- The JSON API lives under `/api` and is mounted **before** the catch-all so API routes are never shadowed.
- `GET /health` → 200 (liveness; no auth).

---

## Sync (v1) → async (later)

- v1 is synchronous for SQLite: `def` handlers, `httpx.Client` in gateways.
- Queries take a `session` parameter, so the eventual async move changes types, not SQL.
- No `asyncio.run()`, no async SQLite.

---

## External calls

- All outbound calls go through a gateway module (httpx), never inline in routes/queries. None at 001; an email-notification gateway arrives at 008; the video gateway (Bunny / Cloudflare) at Phase 2, gated on the Tajikistan playback test.
